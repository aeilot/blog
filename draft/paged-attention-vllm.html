vLLM 与 PagedAttention：把 KV Cache 从一整块内存，改写成一张页表

如果你第一次看 vLLM 这篇论文，可能会有一点误会。

名字叫 PagedAttention，于是人很容易把注意力放在 attention kernel 本身，以为它主要是在改写某种更快的 attention 公式。可真正把文章撑起来的，不是一个更花哨的算子技巧，而是一个很朴素、也很系统的判断：在大语言模型推理里，瓶颈常常不在“算不动”，而在“装不下”。

模型权重当然很大，但它们是静态的，加载进显存之后就一直待在那里。真正麻烦的是另一部分东西：随着请求不断生成新 token 而持续膨胀的 KV cache。它一边增长，一边释放，长度事先又不完全知道。如果你还用“给每个请求预留一整段连续内存”的思路去管它，那么很多显存并不是不够，而是被浪费掉了。

这正是 vLLM 想处理的问题。

先把问题摆正：LLM Serving 为什么会被内存拖住

在 Transformer 的自回归生成里，每生成一个新 token，都要回头看前面所有位置的 key 和 value。为了不在每一步都把前文重新算一遍，系统会把之前各层的 key/value 向量存起来，这就是 KV cache。

这件事本身没有问题。问题在于，它太大，而且增长方式很别扭。

论文里举了一个直观的例子：对一个 13B 级别的模型，模型参数大约已经吃掉了大部分显存，而剩下那一截最关键的空间，往往就是拿来放 KV cache 的。单个 token 的 KV 状态并不小；当 prompt 很长、batch 很大、或者输出继续拉长时，这部分内存会迅速膨胀。于是 serving 系统的 batch size，不再主要受算力限制，而是受显存里还能不能继续塞请求限制。

换句话说，吞吐上不去，很多时候不是 GPU 不够忙，而是 GPU 不敢接更多活。

传统做法为什么笨

过去很多推理系统沿用的是深度学习框架里更自然的一套假设：张量最好是连续存储的。于是，一个请求的 KV cache 往往会被表示成一整块连续 tensor。

听起来很整齐，实际却很浪费。因为请求的长度是动态的。

比如一个请求最多允许生成 2048 个 token，你很可能一开始就按 2048 的长度去预留空间。但最后它也许只生成了 300 个 token，中间那一大段保留位就一直闲着。即便你事先知道它大概不会太长，内存分配器仍然会因为不同大小的连续块反复申请、释放而产生碎片。

论文把这些浪费分成了几类：

  • reserved slots：为了未来可能出现的 token 预留，但此刻用不上。
  • internal fragmentation：一个请求实际长度比预留空间短，块内部空着。
  • external fragmentation：不同请求反复申请和释放，显存被切碎。

更麻烦的是，复杂解码里本来存在共享机会，也被这种“每条序列一整块连续内存”的布局硬生生浪费掉了。

举个例子。假设你对同一个 prompt 做 parallel sampling，采样出 4 个候选回答。它们的前缀 prompt 完全一样，照理说这部分 KV cache 本该只存一份。但如果每个候选序列都被要求拥有自己那块连续空间，那么这份本可共享的前缀状态还是会被复制四遍。

所以问题开始变得清楚：vLLM 要解决的，不是 attention 数学公式错了，而是 KV cache 作为一种“会动态生长的序列状态”，不该再按静态大张量去管理。

PagedAttention 的直觉：别再要求它连续

这篇论文最干净的一步，就是把操作系统里很老的一套想法搬了过来。

在虚拟内存里，程序看到的是连续地址空间，但底层物理页并不需要连续；逻辑页和物理页之间靠页表映射。vLLM 说，KV cache 也可以这么管。

于是，一条序列的 KV cache 不再被看作“一个必须连续的长 tensor”，而被切成固定大小的 KV blocks。你可以把它理解成许多小页：

  • 逻辑上，序列仍然是从左到右连续增长的。
  • 物理上，这些 block 可以散落在显存的不同位置。
  • 系统通过一张 block table 记录：第几个逻辑块，当前映射到哪个物理块，里面已经填了几个 token。

这个改动表面上不大，后果却非常直接。

第一，不需要再为最大长度一次性预留整块空间。序列生成到哪里，就分配到哪里。

第二，因为所有 block 大小一样，外部碎片问题大幅缓解。内部分片也被压缩到一个很小的边界里：最坏情况下，只会浪费最后那个尚未填满的 block。

第三，共享终于变得自然了。两个逻辑块完全可以映射到同一个物理块，只要它们当前看到的是同一段前缀状态。

你会发现，这里最重要的不是“分页”这个词本身，而是它带来的那层间接性。序列不再直接等于显存地址，而是先经过一层映射。正是这层映射，把系统的手脚放开了。

它到底怎么工作

把流程拆开，其实不复杂。

先看一个普通请求。

假设 block size 是 4，一个 prompt 一共有 7 个 token。那系统会先创建两个逻辑块：

  • 逻辑块 0 放前 4 个 token
  • 逻辑块 1 放后 3 个 token，并留下 1 个空位

这两个逻辑块分别映射到某两个物理块上。到这里为止，还没有任何“为 2048 长度预留整段空间”的动作。

随后进入 autoregressive decoding。

如果下一步新生成的 token 还能塞进当前最后一个 block，就直接写进去，把 #filled 加一。等最后一个 block 写满了，再去申请一个新的物理块，并在 block table 里登记这段新映射。

也就是说，序列增长的单位不再是“整体扩容”，而是“按块追加”。

这个设计有一个很重要的结果：一个请求的显存浪费，上限基本被锁在一个 block 里。 这就是为什么论文里会说 vLLM 的 KV cache waste 接近于零。严格来说不是数学意义上的零，而是被压缩到了非常小、非常可控的范围。

PagedAttention 真正改了什么

讲到这里，容易出现第二个误会：既然内存已经分页了，那 attention 计算怎么办？原来不是默认从一整段连续 KV 里顺着读吗？

这就是 PagedAttention 这个名字真正落点所在。

传统 attention 实现默认 key/value 在内存里是连续展开的；而在 vLLM 里，历史 token 对应的 KV 已经被拆散到不同 block 上了。所以 attention kernel 必须学会一件新事：按 block table 去找数据。

它不是先把所有 block 重新拼回一个连续张量，再做 attention；那样就把前面的收益又吐回去了。它做的是：

  1. 先根据当前序列的 block table，知道自己需要访问哪些物理块。
  2. 逐块读取这些 key/value。
  3. 对每个块分别计算 attention score，再把结果归并起来。

所以 PagedAttention 可以理解成一种“接受非连续 KV 布局的 attention 执行方式”。它当然涉及 kernel 实现、访存模式和并行策略，但它的系统意义比算子意义更大。它不是为了让 attention 更炫，而是为了让分页后的 KV cache 还能被高效地读出来。

这一点也顺手解释了一个常见混淆：PagedAttention 不是 FlashAttention。

FlashAttention 主要在解决 attention 计算本身的 I/O 和 SRAM 访问效率，核心是 tiling 与 kernel fusion；PagedAttention 主要在解决 serving 场景里 KV cache 的组织和访问问题，核心是非连续块映射与按块读取。两者都和“attention 更快”有关，但关心的层次并不一样。

它为什么特别适合 vLLM

如果只有分页，这篇论文还不够完整。真正让 vLLM 在工程上站住的，是它把分页、调度和解码策略接在了一起。

1. Parallel Sampling：共享 prompt，按块写时复制

当同一个 prompt 要采样出多个回答时，它们的前缀完全一样。于是最前面的那些逻辑块可以直接映射到同一组物理块上。

这时候系统只需要给这些共享块维护一个 reference count。

如果后续某个样本要继续往最后一个共享块里写新 token，而这个物理块当前还被多个样本共同引用，那么 vLLM 就触发一次 copy-on-write：

  • 先分配一个新物理块
  • 把旧块内容复制过去
  • 让当前样本改写新块
  • 原块的引用计数减一

注意这里的粒度是 block，而不是整条序列。也就是说，复制不会扩散成“把整个 prompt 的 KV cache 全部重拷一遍”，通常只会落在最后那个需要被改写的共享块上。

这很像操作系统里 fork 之后的页级写时复制。不是巧合，论文就是故意这么设计的。

2. Beam Search：共享关系会动态变化

beam search 比 parallel sampling 更有意思，因为它不仅共享 prompt，还会在搜索树内部持续出现“若干候选共享同一前缀”的情况。

如果还沿用旧系统那种每条 beam 拥有独立连续缓存的做法，那么 beam 之间会频繁发生大段 KV copy。候选重组一次，你就搬一次家。

而 vLLM 这里的处理方式要轻很多：

  • 还活着的 beam 继续共享已有前缀块
  • 被淘汰的 beam，引用计数归零后直接释放
  • 新入选的 beam，只在真正需要分叉写入时触发 block 级 copy-on-write

所以 beam search 在 vLLM 上的收益往往比普通采样更明显。因为它共享得更多，也更动态，而分页映射恰好非常适合处理这种“前缀关系不停重组”的访问模式。

这篇论文真正优雅的地方

我觉得这篇论文最值得记住的，不是那句“吞吐提升 2 到 4 倍”，虽然这个结果当然很强。更值得记住的是它处理问题的方式。

很多系统优化会盯着算子、盯着 kernel、盯着 FLOPS 利用率打转。但 vLLM 这篇工作先退了一步，看清楚了 serving 现场真正紧绷的地方在哪里。它说，GPU 不只是一个算力设备，它先是一个有限的内存空间；而 LLM serving 不是一个固定 shape 的离线训练任务，而是一堆长度未知、生命周期交错、还带共享关系的动态序列。

一旦你这样描述问题，操作系统里的 paging、page table、reference count、copy-on-write，几乎就自然地走了进来。

这也是为什么 PagedAttention 读起来会让人觉得很顺。它不是凭空发明了一种玄妙结构，而是把一个已经被很多系统问题验证过的抽象，挪到了 LLM serving 这里。新的部分，在于它把这层抽象接到了 attention kernel、batching 调度、beam search 和 prefix sharing 上，并把整套路径打通。

也别把它神化

当然，这套方法不是没有代价。

分页之后，attention 的访存模式会变复杂，kernel 需要额外查 block table，也要处理变长序列和非连续读取。论文里也坦率给出了这一点：单看 attention kernel 的延迟，PagedAttention 比高度优化的连续实现会有额外开销。

但系统优化常常就是这样。局部算子慢一点，只要换来整体 batch size 更大、显存利用率更高、等待更少，端到端吞吐仍然会明显提升。vLLM 的胜负手不在单次 attention 的极限速度，而在于它终于让更多请求可以同时留在牌桌上。

另外，block size 也不是越大越好。块太小，管理和访存开销上来；块太大，最后一个块的浪费和共享粒度损失又会变严重。论文最后给出的经验值是 block size = 16，本质上是在并行效率和碎片控制之间取一个中间点。

最后再收回来

如果要把 PagedAttention 讲成一句话，我会这样说：

它让大模型推理系统第一次认真对待了这样一个事实: KV cache 不是一块静止的张量，而是一段不断生长、不断分叉、不断回收的内存历史。

一旦你承认它是“历史”，很多事情就会变得不同。你不会再执着于给每个请求铺一条从头到尾的连续道路，而会接受它由许多小块拼起来，接受共享，接受映射，接受在必要的时候复制最后那一页，而不是搬走整座房子。

vLLM 的漂亮，恰恰在这里。它没有改写 Transformer 的基本逻辑，却改写了这些逻辑在显存中的居住方式。很多时候，系统性能的跃迁并不是来自更激进的数学，而是来自一种更合适的组织秩序。
